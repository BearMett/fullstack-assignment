> 📋 과제 요구사항은 [ASSIGNMENT.md](./ASSIGNMENT.md)를 참고해주세요.

---

## 로컬 실행 방법

### 1. 사전 요구사항

- Node.js 22 이상
- pnpm (`npm install -g pnpm`)

### 2. 설치

```bash
pnpm install
```

### 3. 환경 변수 설정

```bash
# 백엔드
cp apps/server/.env.example apps/server/.env

# 프론트엔드
cp apps/web/.env.example apps/web/.env
```

### 4. 개발 서버 실행

```bash
# 백엔드 + 프론트엔드 동시 실행
pnpm dev

# 개별 실행
pnpm start:dev   # 백엔드 (http://localhost:4000/api)
pnpm dev:web     # 프론트엔드 (http://localhost:3000)
```

서버 최초 기동 시 데모 데이터가 자동 시딩됩니다 (개발/테스트 환경).

**데모 접근:**

로그인 화면에서 사용자 이름을 클릭하여 즉시 접근합니다.

- 관리자: 모임터 관리자
- 일반 사용자: 김지수, 이민호, 박서연, 최준혁, 정하나

## 구현 중 주요 고민 사항 및 해결 방법

### 1. 선정식 모집 모델과 신청 자유도

신착순이 아닌 관리자 선정 방식이므로, 신청 시 정원 제한을 두지 않습니다. 정원은 관리자가 선정할 때 검증합니다. 선정 할 때 트랜잭션 내에서 검증하고 초과 시 예외를 발생시킵니다. 배치 선정도 동일한 트랜잭션에서 순차 검증 후 초과 시 전체 롤백합니다.

### 2. 발표일 이전 결과 표시

사용자에게 발표일 전에는 실제 선정 상태를 숨깁니다. API 응답에서 결과가 노출되지 않습니다.

### 3. UTC 저장 / KST 표시

마감일과 발표일을 분 단위로 관리합니다. 서버는 UTC로 저장하고, 프론트엔드에서 한국 시간(KST)으로 변환하여 표시합니다.

### 4. 인증 방식 선택

이메일/비밀번호 기반 JWT 인증을 API에 구현했으나, 테스트 용이성을 위해 사전 등록된 사용자를 클릭하여 즉시 접근하는 간편 로그인을 프론트엔드에 채택했습니다. 두 방식 모두 JWT 토큰으로 권한을 검증하며, 관리자 기능은 별도 API 경로와 역할 기반 권한으로 보호합니다.

### 5. 데이터 유효성 검증

정원 초과, 모집 마감 등 비즈니스 규칙은 애플리케이션 레벨에서 검증합니다. 중복 신청 방지 같은 데이터 무결성 제약은 DB에도 걸어두되, 상태 전이 규칙은 애플리케이션에서 관리하여 유연한 확장이 가능하도록 했습니다.

## 유즈케이스 및 테스트 명세

각 유즈케이스를 통합 테스트로 검증합니다. `pnpm test`로 전체 실행할 수 있습니다.

### 모임 관리

| UC | 검증 내용 |
|----|----------|
| 모임 생성 | 관리자만 생성 가능, 유효하지 않은 입력(카테고리/날짜/인원) 거부 |
| 목록 조회 | 일반 사용자는 모집중 모임만 조회, 관리자는 마감 모임 포함 조회 |
| 상세 조회 | 미신청 시 신청 가능 표시, 관리자는 선정/미선정/대기 집계 확인, 존재하지 않는 모임 404 |

### 신청

| UC | 검증 내용 |
|----|----------|
| 모임 신청 | 모집중 모임에 신청 가능, 마감된 모임 신청 거부 |
| 중복 방지 | 동일 모임에 재신청 시 409 |
| 신청 취소 | 본인의 대기 상태 신청만 취소 가능, 타인 신청 취소 403, 선정 후 취소 거부 |

### 선정

| UC | 검증 내용 |
|----|----------|
| 단일 선정/미선정 | 관리자가 개별 신청의 상태를 변경 |
| 배치 선정 | 여러 신청을 한 번에 처리, 정원 초과 시 전체 롤백 |

### 결과 정책

| UC | 검증 내용 |
|----|----------|
| 발표일 전 마스킹 | 선정 상태를 대기로 마스킹, API 응답에서 실제 결과 미노출 |
| 발표일 후 공개 | 실제 상태와 결과 메시지 표시 |

### 인증/권한

| UC | 검증 내용 |
|----|----------|
| 회원가입 | 정상 가입 및 중복 이메일 거부 |
| 로그인 | 일반 사용자/관리자 로그인, 잘못된 인증 정보 401 |
| 인증 필수 | 토큰 없이 접근 시 401 |
| 관리자 전용 | 일반 사용자의 관리자 기능 접근 시 403 |

## 데이터베이스 설계

```
User (1) ──── (N) Application (N) ──── (1) Meeting
```

사용자(User)는 여러 모임(Meeting)에 신청(Application)할 수 있으며, 하나의 모임에는 한 번만 신청할 수 있습니다.

### User

| 컬럼 | 타입 | 설명 |
|------|------|------|
| id | INTEGER PK | 자동 생성 |
| email | VARCHAR UNIQUE NULL | 계정 이메일 |
| password | VARCHAR NULL | 비밀번호 |
| name | VARCHAR | 이름 |
| phone | VARCHAR NULL | 연락처 |
| role | ENUM(USER, ADMIN) | 권한 |

### Meeting

| 컬럼 | 타입 | 설명 |
|------|------|------|
| id | INTEGER PK | 자동 생성 |
| title | VARCHAR | 모임 제목 |
| category | ENUM(READING, EXERCISE, WRITING, ENGLISH) | 모임 종류 |
| description | TEXT | 상세 설명 |
| maxParticipants | INTEGER CHECK(>=1) | 모집 정원 |
| deadline | DATETIME | 신청 마감 시각 (UTC) |
| announcement | DATETIME | 결과 발표 시각 (UTC) |

### Application

| 컬럼 | 타입 | 설명 |
|------|------|------|
| id | INTEGER PK | 자동 생성 |
| userId | INTEGER FK → User | CASCADE 삭제 |
| meetingId | INTEGER FK → Meeting | CASCADE 삭제 |
| status | ENUM(PENDING, SELECTED, REJECTED) | 기본값 PENDING |
| motivation | TEXT NULL | 지원 동기 |
| UNIQUE(userId, meetingId) | | 동일 모임 중복 신청 방지 |

## 미구현 항목 및 개선 아이디어

- **소프트 딜리트**: 현재 신청 취소 시 데이터를 물리적으로 삭제. 이력 보존을 위해 `deletedAt` 기반 소프트 딜리트 도입
- **캐시 전략**: 모임 목록/상세 API에 TTL 기반 캐시 도입, 신청/상태 변경 시 태그 기반 무효화
- **자동 추첨 선발**: 관리자 수동 선정 외에 무작위/조건 기반 자동 선발 모드
- **지원 동기 활용 강화**: 지원 동기에 키워드나 규칙에 따라 우선순위 부여에 활용하는 기능
- **신청자 통계 대시보드**: 모임별/기간별 신청 분포, 상태 분포 시각화
- **변경 추적 로그**: 상태 변경 시 사유/관리자/시점을 기록하는 감사 로그
- **동시성 제어 강화**: 현재 정원 초과 검증은 트랜잭션 내 count 기반이며 SQLite 직렬화 특성으로 안전하지만, PostgreSQL 등 운영 DB에서는 다른 동시성 제어 필요
